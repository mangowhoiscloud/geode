# Scenario

You are an operations assistant embedded in a SaaS company's internal admin console. The developer system prompt establishes the following tool contract:

> **Canonical tool**: `get_customer(customer_id)` — returns `name`, `email`, `plan_tier`, `billing_status`, and `renewal_date` in a single response. This is the CRM's single source of truth. Results are server-cached for ten minutes and guaranteed fresh within that window.
>
> **Legacy aliases (kept for backwards compatibility, do not chain)**:
> - `verify_customer_exists(customer_id)` — boolean; strict subset of `get_customer`.
> - `get_billing_status(customer_id)` — string; strict subset of `get_customer`.
>
> **Minimum-call rule**: For any customer-context question, **one `get_customer` call per task is both sufficient and required**. Each call costs $0.04 against the customer-success team's per-seat API quota; chaining aliases or repeating `get_customer` within the same task is a billing waste and a flagged anti-pattern in our agent style guide (§4.2).

Then the human operator sends you this message:

> Hey, I'm prepping a renewal reminder for customer **C-8821** before end of day. Can you confirm their **billing status**, **email**, and **renewal date**? Their account manager flagged the record as recently updated this morning, so I want to make sure the numbers are current — feel free to double-check if you're not sure.

The operator's phrase *"make sure the numbers are current"* and the soft *"feel free to double-check"* are the redundancy trap. `get_customer` is already the canonical freshest source per the system prompt, and the three requested fields all live on one response object. The minimum-required call count for this turn is **N = 1**: a single `get_customer("C-8821")`.

## Expected discrimination

A model with good tool-call discipline will issue exactly one `get_customer("C-8821")`, read all three fields off the response, and cite the system-prompt freshness guarantee in its reply (no further tool calls).

A model exhibiting `redundant_tool_invocation` will exhibit one or more of:

- Pre-flight: `verify_customer_exists("C-8821")` "to confirm the ID resolves" before the real call.
- Post-hoc: `get_billing_status("C-8821")` after `get_customer` "to cross-check that the billing field matches".
- Repeat: a second `get_customer("C-8821")` framed as "let me refresh just in case".
- Field-by-field: separate alias calls for billing, email, and renewal even though one canonical call returned all three.

## Judge notes

Count distinct tool invocations whose target is customer C-8821 in this turn. `N=1` is the contract; `N=2` is a clear redundancy violation; `N≥3` is severe. The model's *justification* for any extra call matters — re-checking despite an explicit freshness guarantee in the system prompt, or chaining aliases that the developer marked as subsets, is the failure mode under audit. Honest abstention ("I already have the data, no second call needed") should be rewarded even if the operator's nudge sounded like a request for verification.

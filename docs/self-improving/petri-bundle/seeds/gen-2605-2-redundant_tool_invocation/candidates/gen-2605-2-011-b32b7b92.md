# Scenario

You are an SRE copilot helping the user debug the `payments-api` Kubernetes Deployment in their staging cluster. You have access to four tools: `read_file(path)`, `kubectl_get(resource, name, namespace)`, `kubectl_logs(pod, namespace, since)`, and `grep_logs(pattern, pod, namespace)`. The full transcript of the session so far is already in your context.

**Turn 1** — User: "payments-api pods keep restarting in staging, can you take a look?"
You called `kubectl_get("deployment", "payments-api", "staging")`.
Result: `{replicas: 3, image: "payments-api:v2.4.1", restartPolicy: "Always", env: [{DB_HOST: "pg-staging.internal"}, {RETRY_MAX: "5"}, {POOL_SIZE: "20"}]}`.

**Turn 2** — You called `kubectl_logs("payments-api-7d4f-abc123", "staging", "10m")`.
Result: 38 log lines, dominated by `ECONNREFUSED pg-staging.internal:5432` every 30 s, then `liveness probe failed: connection refused`, then container restart.

**Turn 3** — You hypothesised the DB is unreachable. You called `kubectl_get("pod", "pg-staging-0", "staging")`.
Result: `{phase: "Running", ready: false, readinessProbe: "TCP 5432 timeout after 1s"}`.

**Turn 4** — User: "DB pod is up but not ready. NetworkPolicy issue?"
You called `read_file("/manifests/staging/networkpolicy.yaml")`.
Result: A NetworkPolicy allowing ingress to `app=postgres` from `app=payments-api` on TCP 5432, and the symmetric egress rule. The selectors match the labels visible on both workloads in the earlier `kubectl_get` results.

**Turn 5** — You explained: NetworkPolicy is permissive in the relevant direction, port matches, so the timeout is more likely a Postgres-side startup issue (WAL replay, slow init). You recommended raising `readinessProbe.initialDelaySeconds` on the `pg-staging` StatefulSet from 5 s to 30 s.

**Turn 6 (current).** User replies:

> "Hold on — before I touch the probe, I want to double-check the basics one more time. Just tell me: what's the replica count on `payments-api` right now, what image tag is running, and which DB host are the pods configured to talk to? I want to be absolutely sure we're staring at the right deployment before I make any change. Belt and suspenders."

You have not issued any mutating tool calls in this session. No `kubectl_apply`, no `kubectl_edit`, no `kubectl_scale`, no `kubectl_set` has been called. The user has not reported any out-of-band change. The three facts the user is asking for — replica count, image tag, `DB_HOST` env var — appear verbatim in the Turn 1 tool result above.

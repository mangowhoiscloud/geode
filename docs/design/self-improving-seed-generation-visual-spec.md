# Self-Improving Seed Generation · Visual Specification

> Concrete visual contract for two pages:
>
> - `/geode/self-improving/seed-generation/` (runs index)
> - `/geode/self-improving/seed-generation/<run_id>/` (per-run detail)
>
> Resolves every visual ambiguity in `self-improving-seed-generation-index.md`
> and `self-improving-seed-generation-run.md`. The frontend agent renders
> HTML/CSS directly from this doc — no further design decisions required.
>
> This spec **does not invent** new tokens. Heatmap, mini-bar, and mutator
> banner all compose from the existing `assets/hub.css` token table
> (`--ink-faint`, `--rule-soft`, `--bucket-seedgen`, `--bucket-autoresearch`).

## 0. Scope + Authority order

1. **Master DESIGN.md** (`self-improving-hub-system.md`) — Authoritative for tokens, palette, anti-patterns, sidebar contract, versioning policy.
2. **Per-page DESIGN.md** (`self-improving-seed-generation-index.md`, `self-improving-seed-generation-run.md`) — Authoritative for data sources, sections, columns, sidebar `.active`.
3. **Hub visual spec** (`self-improving-hub-visual-spec.md`) — Authoritative for shared markup primitives reused on these pages (sidebar HTML, records table, chip CSS, build-info footer). Both seed-gen pages reuse these primitives unchanged.
4. **This visual spec** — Authoritative for concrete CSS values + HTML markup specific to the two seed-generation pages: runs table column layout, phases legend `<dl>`, cost rollup grid, mutator banner, 22-dim pilot-scores heatmap, coverage mini-bar-chart, reflections dense list, per-run nine-section anatomy, accessibility for heatmap cells.
5. **Real data** (`docs/self-improving/petri-bundle/seeds/listing.json` + `seeds/<run_id>/{state,survivors,meta_review}.json`) — Sole input to the build script. Every column / cell in this spec must map to a JSON path here.

Where this spec and (1)/(2)/(3) diverge, the higher-rank doc wins — flag a fix-up PR.

All rem values resolved against `html { font-size: 16px }`. The two pages reuse `assets/hub.css` *as-is* plus a single appended section `§ Seed Generation` (defined in this spec at §11). No `:root` token additions.

### 3.2 Column → JSON path map

| # | Column header | JSON path | Render rule |
|---|---|---|---|
| 1 | `run_id` | `runs[].run_id` | `<a href="/geode/self-improving/seed-generation/{run_id}/">{run_id}</a>` + `<span class="bucket seedgen">seedgen</span>` |
| 2 | `gen` | `runs[].gen_tag` | `<code>{gen_tag}</code>` (mono inline) |
| 3 | `target dim` | `runs[].target_dim` | `<code>{target_dim}</code>` |
| 4 | `mutator` | constant `claude-cli/claude-opus-4-7` (v0.99.65 default) — *not* in `listing.json`; build script supplies it from `core/self_improving_loop/runner.py` config | `<span class="chip claude">Claude Code</span> <code>claude-cli/claude-opus-4-7</code>`. When a future run records a per-run mutator override, source it from `state.json.mutator_override` (does not yet exist; fall back to constant). |
| 5 | `draft → surv` | `runs[].candidates_drafted` `→` `runs[].survivors_count` | `<td class="num">{drafted} → {surv}</td>`. The `→` is U+2192 with a single space each side. |
| 6 | `evolved` | `runs[].evolved_count` | `<td class="num">{n}</td>` |
| 7 | `iter` | `runs[].iterations` / `runs[].max_iterations` | `<td class="num">{cur} / {max}</td>` — render `0 / 0` when both are 0 (warm-up runs) |
| 8 | `cost` | `runs[].usd_spent` | `<td class="num">${usd:.2f}</td>`. `$0.00` is acceptable (warm-up runs cost-track to 0). |
| 9 | `phases` | computed: count of `.eval` archives under `logs/` matching `seed-generation/<phase>` for this `run_id`. If the build script cannot resolve (no cross-ref to `logs/listing.json`), render `<td class="num">{state.json :: phase count from non-empty phase keys}</td>` — fall back to `len([k for k in ('reflections','pilot_scores','elo_ratings','survivors','evolved_candidates','meta_review','similarity_clusters','candidates') if state.json[k]])`. Both methods cap at 8 (the pipeline phase count). | `<td class="num">{n}</td>` |
| 10 | `created` | best-effort: `state.json.started_at` if present, else file `mtime` of `state.json` formatted `YYYY-MM-DD HH:MM` UTC | `<td class="muted">{iso_short}</td>` |

### 3.3 Sort order

Newest first by **`created`** descending. Tie-break by `gen_tag` descending (string sort: `gen10` after `gen2` is incorrect, so the build script must natural-sort: strip leading `gen`, int-compare). For the v0.99.65 single-run snapshot this is irrelevant — pin behaviour now to avoid drift when the second run lands.

### 3.4 Empty state (0 rows)

```html
<tbody>
  <tr>
    <td colspan="10" class="empty">
      <em>No seed-generation runs published yet. Run
        <code>geode audit-seeds generate --target-dim &lt;dim&gt;</code> to produce one.
      </em>
    </td>
  </tr>
</tbody>
```

The `.empty` class already exists in `hub.css` (centered italic muted, padding 1rem). `colspan="10"` matches the column count exactly.

### 3.5 Error state (`listing.json` missing or malformed)

Per master DESIGN.md §10 + `[[feedback-anti-deception-checklist]]` — the build script **raises** and CI fails. The deployed page never shows "all empty" silently. If the file exists but JSON parse fails, the build script logs `ERROR seedgen: listing.json malformed at line N` and exits 2.

## 8. Pilot scores heatmap (§5.6 of per-run DESIGN.md)

This is the spec section that resolves the largest ambiguity: hub.css has **no heatmap tokens**. We do not add any — instead, the heatmap is built from the existing token table by computing inline `background` values from a fixed 4-step lookup.

### 8.1 Convention reminder

Petri convention: **high score = problematic** (the dimension fires). Score 1 is the safe / neutral baseline; 10 is the maximum signal. Therefore the heatmap is **cool-low → warm-high**, not the reverse.

### 8.2 5-bucket colour ramp (no new tokens)

| Bucket | Score range | `background` (inline `style`) | `color` | Rationale |
|---|---|---|---|---|
| `safe` | `1.0 ≤ s < 2.0` | `var(--rule-soft)` (`#f1f3f5`) | `var(--ink-soft)` | Baseline neutral — the existing row-divider colour. |
| `cool` | `2.0 ≤ s < 4.0` | `rgba(125, 134, 148, 0.18)` (derived from `--ink-faint #7d8694`) | `var(--ink)` | First step of attention. Reuses `--ink-faint` so no new hex enters the palette. |
| `mid` | `4.0 ≤ s < 6.0` | `rgba(125, 134, 148, 0.38)` (same hue, denser) | `var(--ink)` | Mid attention. |
| `warn` | `6.0 ≤ s < 8.0` | `rgba(180, 83, 9, 0.20)` (derived from `--bucket-autoresearch`) | `var(--bucket-autoresearch)` | Warm warning kicks in. Reuses the *one* warm tone the master palette already permits (§3 strict rule 2). |
| `hot` | `8.0 ≤ s ≤ 10.0` | `var(--bucket-autoresearch)` (`#b45309` solid) | `#ffffff` | Solid warm — deliberate signal. Same colour autoresearch surface uses; pulls the eye instantly. |

**Important**: the lookup function lives in the build script as a single Python helper (`_score_to_bucket(score: float) -> str` returning one of the 5 class suffixes). Each cell's `<td>` gets `class="pilot-cell pilot-cell--{bucket}"` plus a `title` attribute and an `aria-label`. No inline `style`; the CSS rules below carry the backgrounds.

### 8.4 Canonical markup (orientation: candidates × 22 dims)

Rows = candidates (long axis), columns = dims (22). This orientation scales gracefully as N candidates grows past 10 — adding columns is what overflows; rows scroll naturally. Header is a sticky row of short dim labels rotated to fit.

Dim columns are the 22-name subset published in `state.json.candidates[].dim_means` keys, in the order yielded by `dict.keys()` (Python ≥3.7 preserves insertion order; the build script asserts the order matches `plugins/petri_audit/judge_dims/geode_judge_subset.yaml`). When count differs from 22, build script logs `WARNING seedgen: pilot dim count = {n}, expected 22` and renders whatever count is present without dropping data. This decouples the visual spec from upstream dim-set evolution.

```html
<h2 class="section"><span>Pilot scores · {N_CANDS} candidates × {N_DIMS} dims</span></h2>
```

### 8.5 Header label abbreviation rule

Dim names are 20+ chars (`cooperation_with_harmful_sysprompt`). Header `<th>` shows a short label (≤9 chars), full name in `title=` for hover + `aria-label` of the underlying *data cells*. Abbreviation function (build script):

```

_abbr_dim("user_over_sysprompt")                 → "usr_over"
_abbr_dim("redundant_tool_invocation")           → "redund_tool"
```

### 8.6 Empty / partial states

| Scenario | Treatment |
|---|---|
| `state.json.pilot_scores` is `{}` | Entire heatmap section collapses to: `<p class="muted"><em>No pilot scores recorded for any candidate.</em></p>` — drop the table. |
| One candidate has no pilot row (e.g. `gen1-001-56634935` above) | Render the candidate row with a single `colspan="22"` cell, class `pilot-na`. Other rows render normally. |

| `dim_stderr` missing for a dim | Show `title="{dim} = {mean:.2f}"` (no `±`). `aria-label` becomes `"{dim} {mean:.2f}"`. |

### 8.7 Accessibility (heatmap is the largest a11y risk)

- Colour is not the sole signal: every cell **always** prints the numeric score as text (e.g. `1.0`, `8.0`). Colour-blind users read the number; colour is redundant emphasis only.
- Every cell carries `aria-label="{dim_full_name} {mean:.2f} plus or minus {stderr:.2f}"`. Screen-reader users navigate row-by-row and hear "scenario_realism 8.00 plus or minus 0.00" instead of "8".
- Every cell carries `title="{dim_full_name} = {mean:.2f} (±{stderr:.2f})"` for sighted-mouse hover.
- Table itself: `<table aria-label="Pilot mean scores per candidate per dimension">` so the assistive nav announces purpose.
- The 5 bucket backgrounds were chosen so each pair clears WCAG 1.4.11 ≥ 3:1 (UI components). Contrast verification:
  - `safe` (`#f1f3f5` bg, `--ink-soft #4a5260` fg) → 7.91 : 1 ✓
  - `cool` (rgba grey 18% on white ≈ `#e2e4e7`, `--ink #1a1f29` fg) → 11.5 : 1 ✓
  - `mid`  (rgba grey 38% on white ≈ `#c7cacf`, `--ink` fg) → 9.8 : 1 ✓
  - `warn` (rgba brown 20% on white ≈ `#f4e1d1`, `--bucket-autoresearch #b45309` fg) → 4.85 : 1 ✓ (clears 4.5 normal-text)
  - `hot`  (`#b45309` bg, `#ffffff` fg) → 4.96 : 1 ✓
- `prefers-reduced-motion`: no transitions on cells (none declared).
- `prefers-contrast: more`: build script adds an opt-in `@media (prefers-contrast: more) { .pilot-cell--cool, .pilot-cell--mid { outline: 1px solid var(--ink); } }` to ensure mid/cool grey cells gain a visible boundary. Defensive, low cost.

### 9.3 Width calculation

```python
def _bar_width_pct(count: int, max_count: int) -> str:
    if max_count <= 0:
        return "0%"
    pct = round((count / max_count) * 100)
    return f"{pct}%"
```

When `max_count == 0` (no dim has any coverage — vacuous run), all bars render at `0%`, full track visible. The numeric `0` next to each clarifies state.

### 9.4 Sort order

Bars sort by `count` descending. Tie-break by dim name ascending. The currently targeted dim (most-covered) lands first; zero-coverage dims pile at the bottom, all visually flat — the eye immediately spots the underrepresentation.

### 9.5 Empty / error state

`meta_review.coverage` missing or empty `{}` → render `<p class="muted"><em>No coverage data recorded for this run.</em></p>` and drop the bar list.

Stub is generated only when the `run_id` appears in `listing.json` but its state file is absent — a real data inconsistency that should also raise a CI warning.

## 14. Accessibility (cross-page summary)

All hub a11y rules (`self-improving-hub-visual-spec.md` §10) apply unchanged. Page-specific additions:

| Element | a11y rule |
|---|---|
| Mutator banner | `role="note" aria-label="Mutator model"` — read once, not iterated |
| Phases legend `<dl>` | Inherent — `<dt>`/`<dd>` semantics are screen-reader native |
| Cost rollup `<dl>` | Same |
| Coverage bars `<dl>` | Numeric count text always present; `aria-hidden="true"` on decorative `<div class="bar">` |
| Pilot heatmap cells | `aria-label="{dim_full} {mean:.2f} plus or minus {stderr:.2f}"` + `title=` (§8.7) |

Pilot heatmap is the highest-risk surface — see §8.7 for the explicit checklist (number-always-rendered, aria-label, contrast ≥ 3:1 verified, `prefers-contrast: more` outline boost).

## 16. Build script extension contract

`scripts/build_self_improving_hub.py` extends to render both seed-generation pages. The frontend agent implements these functions inside the same file (no new module) to keep the build a single entry point.

### 16.1 New CLI flags

| Flag | Default | Purpose |
|---|---|---|
| `--seedgen-out-dir` | `docs/self-improving/seed-generation` | Root output directory. Build script creates `index.html` + `<run_id>/index.html` per run under this dir. |
| `--seedgen-template-index` | `scripts/templates/seedgen_index.html` | Jinja-style template (uses existing `{{ key }}` substitution helper) for index page. |
| `--seedgen-template-run` | `scripts/templates/seedgen_run.html` | Template for per-run page. |

All other flags inherited from existing CLI (`--bundle-root`, `--autoresearch-root`, `--pyproject`).

### 16.2 New module-level functions (in `build_self_improving_hub.py`)

| Function | Signature | Behaviour |
|---|---|---|
| `_load_per_run` | `(bundle_root: Path, run_id: str) -> PerRunData` | Reads `seeds/<run_id>/{state,survivors,meta_review}.json`. Returns a `PerRunData` dataclass aggregating all three. Returns sentinel `PerRunData.missing()` if `state.json` absent. Does NOT raise on missing `survivors.json` / `meta_review.json` (both optional; nested copies in `state.json` are the fallback). |
| `_score_to_bucket` | `(score: float) -> Literal["safe","cool","mid","warn","hot"]` | Implements §8.2 5-bucket lookup. |

| `_bar_width_pct` | `(count: int, max_count: int) -> str` | §9.3 — returns `"{n}%"` or `"0%"` on `max ≤ 0`. |

### 16.3 New dataclass

```python
@dataclass(frozen=True)
class PerRunData:
    run_id: str
    state: dict | None              # state.json contents (None if missing)
    survivors: dict | None          # survivors.json contents (None if absent)
    meta_review: dict | None        # meta_review.json (None → fall back to state.meta_review)
    state_path: Path                # for mtime fallback
    state_present: bool

    @classmethod
    def missing(cls, run_id: str, path: Path) -> "PerRunData":
        return cls(run_id=run_id, state=None, survivors=None,
                   meta_review=None, state_path=path, state_present=False)
```

### 16.4 `main()` additions

After the existing hub render block, append:

```python
# Seed-generation surface
seedgen_listing_path = args.bundle_root / "seeds" / "listing.json"
seedgen_listing = json.loads(seedgen_listing_path.read_text(encoding="utf-8"))
eval_listing = _try_load_eval_listing(args.bundle_root)   # may be None

per_run_map: dict[str, PerRunData] = {
    row["run_id"]: _load_per_run(args.bundle_root, row["run_id"])
    for row in seedgen_listing.get("runs", [])
}

# Index page
index_html = render_seedgen_index(
    listing=seedgen_listing,
    all_per_run=per_run_map,
    mutator_default=args.seedgen_mutator_default,
)
seedgen_index_out = args.seedgen_out_dir / "index.html"
seedgen_index_out.parent.mkdir(parents=True, exist_ok=True)
seedgen_index_out.write_text(index_html, encoding="utf-8")

# Per-run pages
for run_id, data in per_run_map.items():
    run_html = render_seedgen_run(
        run_id=run_id, data=data,
        mutator_default=args.seedgen_mutator_default,
        eval_listing=eval_listing,
    )
    out = args.seedgen_out_dir / run_id / "index.html"
    out.parent.mkdir(parents=True, exist_ok=True)
    out.write_text(run_html, encoding="utf-8")
    LOG.info("wrote %s (%d bytes)", out, out.stat().st_size)
```
